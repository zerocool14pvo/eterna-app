# Инструкции для агентов (AI)

## Язык

- Пишем **только на русском**: комментарии в коде, сообщения коммитов по запросу, пояснения в PR — на русском.
- Отвечаем пользователю **только на русском**.

## Рабочий цикл после задачи (git и следующий шаг)

После того как **задача выполнена** и пользователь пишет, что сделал её / проверит / проверил:

1. **Спросить явно**, нужно ли **запушить** текущие изменения в **git** (например: «Запушить изменения в git?»).
2. Если пользователь ответил **да** (или однозначно согласился): при необходимости оформить **commit** с осмысленным сообщением **на русском**, затем выполнить **`git push`** (если настроен `remote` и это уместно; при ошибках доступа — сообщить, не ломая историю).
3. **После** push (или если пользователь отказался от push) — **спросить**, какая **следующая задача**.
4. Для **каждой** новой задачи повторять тот же цикл: выполнить → дождаться реакции пользователя → спросить про push → при согласии push → спросить про следующую задачу.

Не пушить и не делать коммит **без явного согласия** пользователя, если только он сам не попросил «закоммить и запушь» в формулировке задачи.

## TypeScript

- **Запрещено** использовать тип `any`. Используйте `unknown`, дженерики, union-типы, `satisfies`, явные интерфейсы и типы из библиотек.

## Стек проекта

- **React** (UI), **Vite** (сборка и dev-сервер), **TypeScript**, **SCSS**, линтинг и форматирование по конфигам репозитория.
- Цель по производительности: **максимально возможная оптимизация** на уровне бандла, рендера, сети и кешей — без преждевременной микрооптимизации «ради самой оптимизации»; тяжёлые решения обосновывайте (размер бандла, LCP, повторные запросы).

## Импорты и алиасы (TS / TSX)

Настроены в **`vite.config.ts`** и **`tsconfig.app.json`**:

| Префикс        | Каталог           | Пример                                                             |
| -------------- | ----------------- | ------------------------------------------------------------------ |
| `@/`           | `src/`            | `import { x } from '@/types/withClassName'`                        |
| `#components/` | `src/components/` | `import { Card } from '#components/Card'`                          |
| `#styles/`     | `src/styles/`     | `import '#styles/reset.scss'` (при необходимости отдельного входа) |
| `#assets/`     | `src/assets/`     | `import logo from '#assets/logo.svg'`                              |
| `#screens/`    | `src/screens/`    | `import { Home } from '#screens/Home'`                             |

Префикс **`#`** для папок вместо второго `@`, чтобы путь не выглядел как npm-scope (`@scope/pkg`) и **Prettier** корректно относил импорт к своим группам, а не к сторонним пакетам.

Глобальный **сброс стилей** подключается из **`index.scss`** через `@use './styles/reset.scss'`, чтобы порядок слоёв CSS не зависел от сортировки импортов в `main.tsx`.

Порядок импортов в TS/TSX задаёт **Prettier** и плагин **`@ianvs/prettier-plugin-sort-imports`** (см. `.prettierrc`): встроенные модули Node → npm → `#components` / `#styles` / `#assets` / `#screens` → `@/` → относительные `.ts`/`.tsx` → в конце `*.scss` / `*.css`.

## Производительность, кеширование и ассеты

### React

- Дробите приложение: **`React.lazy` + `Suspense`** для редко открываемых экранов и тяжёлых страниц; тяжёлые библиотеки подключайте через **динамический `import()`**, а не статически в корне, если это не критично для первого экрана.
- Избегайте лишних **перерисовок**: стабильные колбэки (`useCallback`) и мемоизация (`useMemo`, `memo`) — только там, где есть измеримая выгода или дорогой дочерний список; не оборачивайте всё подряд.
- Списки: стабильные **`key`**, виртуализация при очень длинных списках.
- Не создавайте в рендере новые объекты/массивы в пропсах без нужды (они ломают мемоизацию детей).

### Изображения и медиа

- У **`img`** задавайте **`width` и `height`** (или явные размеры в вёрстке), чтобы уменьшить CLS.
- Для контента ниже первого экрана — **`loading="lazy"`**; для LCP-изображения — **`fetchPriority="high"`** (и без `lazy`).
- По возможности — современные форматы (**WebP / AVIF**) и **`srcset` / `<picture>`** под плотность и ширину; не тяните в бандл гигантские растровые файлы, если их можно отдать статикой/CDN.
- **`decoding="async"`** для некритичных картинок — по ситуации.
- Иконки и простая графика — **SVG** или спрайт, чтобы не плодить десятки HTTP-запросов к мелким PNG.
- Кеширование в браузере для картинок обеспечивается **долгим `Cache-Control` на CDN/сервере** для имён с **хэшем в URL** (у статики из production-сборки Vite хэш уже в имени файла — такие URL можно помечать как immutable).

### Статика, бандл и HTTP-кеш

- Сборка **production** должна оставаться **разделённой на чанки** (Vite по умолчанию); не отключайте сплиттинг без причины.
- Новые тяжёлые зависимости проверяйте по размеру (**анализ бандла** при сомнениях).
- Для данных с API (когда появятся) — осмысленные заголовки кеша на сервере и на клиенте библиотеки вроде **TanStack Query / SWR** с **`staleTime` / инвалидацией**, чтобы не дублировать одинаковые запросы и уважать кеш.

### Инструкции для ИИ-агентов при правках кода

- Сохраняйте и усиливайте **дружелюбность к кешу**: стабильные URL для статики, не инлайнить огромные base64 без запроса пользователя.
- При добавлении изображений — сразу продумывайте **lazy / приоритет / размеры / формат**.
- Не подключайте критичный для FCP/LCP код в **главный чанк** без необходимости; выносите за `lazy` / динамический импорт.
- Если задача касается «максимальной оптимизации», явно учитывайте **LCP, CLS, TBT** и размер **JS/CSS**, а не только «чтобы работало».

## Стиль объявления компонентов

- Все React-компоненты объявляйте **стрелочными функциями** и экспортируйте константу (например `const UserCard = (props: UserCardProps) => { ... }`), без `function ComponentName() {}`.

## Пропсы компонентов в `src/components`

- Любой компонент из каталога **`src/components`** (импорты с алиасом `@/components/...`) обязан принимать в пропсах **как минимум** `className?: string` для корневого элемента и **склеивать** его с BEM-классами через `@bem-react/classname` (см. ниже).
- Базовый тип: `WithClassName` из `@/types/withClassName`. Интерфейс пропсов: `export type UserCardProps = WithClassName & { title: string }`.
- Пример корня: `className={cnUserCard('Root', className)}` — второй аргумент — внешний `className` (может быть `undefined`, лишние пробелы не попадут в строку).

## BEM и `@bem-react/classname`

Используем библиотеку **`@bem-react/classname`** и [соглашение React/BEM](https://en.bem.info/methodology/naming-convention/#react-style): элемент через дефис, модификатор через `_`.

### Константа `cn` на блок

Для каждого блока заводите **одну** фабрику с префиксом `cn` и именем блока (как в коде компонента), **на верхнем уровне модуля** (не внутри компонента):

```ts
import { cn } from '@bem-react/classname'

const cnApp = cn('App')
const cnUserCard = cn('UserCard')
```

Имя блока в строке совпадает с именем React-компонента / папки: `UserCard` → `cn('UserCard')`, не `cn('user-card')`.

### Элементы и вложенность

- Блок: `cnApp()` → `App`.
- Элемент одного уровня: `cnApp('Center')` → `App-Center`.
- Цепочка «элемент внутри элемента» в React-пресете задаётся **одним** именем с дефисами: `cnApp('Hero-Base')` → `App-Hero-Base` (не вызывайте `cn('App', 'Hero')('Base')` — так библиотека не строит цепочку).

### Модификаторы

Первый аргумент — объект с булевыми или строковыми значениями:

```ts
cnUserCard({ size: 'm', active: true })
// UserCard UserCard_size_m UserCard_active
```

### Смешивание классов (миксы)

Внешние классы (проп `className` и другие строки-миксы) передавайте **вторым аргументом** после имени элемента (или после объекта модификаторов у блока):

```ts
cnUserCard('Title', className)
```

Второй аргумент — **mix**: строка или **массив строк**, если нужно склеить несколько классов:

```ts
cnUserCard('Header', ['extra-layout', className].filter(Boolean))
```

`undefined` и пустые строки в массиве отфильтруйте, чтобы не попадали в `className` DOM.

### Стили (SCSS)

- **Без CSS Modules**: используйте обычные файлы **`ИмяКомпонента.scss`**, подключение в TSX — `import './ИмяКомпонента.scss'`.
- Селекторы **повторяют** строки из `cn`: блок `.UserCard`, элемент `.UserCard-Title`, модификатор `.UserCard_size_m` и т.д. Инкапсуляция за счёт префикса блока (имя компонента), а не за счёт хэшей модулей.
- Общие листы (`index.scss`, `reset.scss`) трогают классы компонентов только осознанно (редкие исключения).

### Запреты

- Не использовать произвольные строки классов в `className` без `cn` (кроме редких технических случаев).
- Не объявлять `const cnBlock = cn('Block')` внутри тела компонента — только на уровне модуля.

## Структура React-компонентов

Каждый компонент живёт в **своей папке**, имя папки совпадает с именем компонента в **PascalCase** (например `UserCard`).

Внутри папки обязательно:

| Файл                 | Назначение                                                   |
| -------------------- | ------------------------------------------------------------ |
| `ИмяКомпонента.tsx`  | Реализация компонента                                        |
| `ИмяКомпонента.scss` | Стили компонента (обычный SCSS, **не** CSS Modules)          |
| `index.ts`           | Реэкспорт: `export { ИмяКомпонента } from './ИмяКомпонента'` |

Импорт снаружи — через папку (баррель), без `.index` в пути, например:

```ts
import { UserCard } from '@/components/UserCard'
```

Не кладите несколько несвязанных компонентов в одну папку. Общие утилиты — в `utils/`, общие типы — в `types/`, общие константы — в `constants/`.

## Страницы (`src/screens`)

- Все **маршрутизируемые страницы** (экраны верхнего уровня под `react-router-dom`) живут в **`src/screens`**, каждая — в **своей папке** **PascalCase** по имени экрана, с тем же набором файлов, что у компонента: **`ИмяЭкрана.tsx`**, **`ИмяЭкрана.scss`**, **`index.ts`**.
- Импорт экрана: `import { Home } from '#screens/Home'` (баррель из папки экрана).
- На корневом элементе экрана — проп **`className?: string`** и **`@bem-react/classname`** (как у компонентов в `src/components`), чтобы при необходимости оборачивать экран извне.
- **`App.tsx`** только собирает **`<Routes>`** и подключает экраны из `#screens/...`; разметку страниц не размещайте в `App.tsx`, кроме оболочки приложения (при необходимости).
